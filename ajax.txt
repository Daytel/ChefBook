Сценарий A — Комментарии к рецепту (AJAX: загрузка + добавление)
Контекст

Страница: recipe.html.
Есть блок комментариев: список, поле ввода, кнопка «Отправить», кнопка «Ещё».

Цель

При открытии страницы подгрузить первые 10 комментариев без перезагрузки.

Позволить добавлять комментарий через AJAX.

Поддержать пагинацию кнопкой «Ещё».

UI-элементы

#commentList — контейнер комментариев

#commentBox — textarea

#postCommentBtn — кнопка отправки

#loadMoreComments — кнопка подгрузки следующей порции

#commentSpinner (опционально) — индикатор загрузки

API
1) Получение комментариев

GET /api/recipes/:recipeId/comments?offset=0&limit=10

Ответ 200 OK:

{
  "recipeId": "42",
  "comments": [
    { "id": 11, "user": "ivan", "text": "Класс!", "createdAt": "2026-02-16T10:00:00Z" }
  ],
  "more": true
}

2) Добавление комментария

POST /api/recipes/:recipeId/comments

Тело:

{ "text": "Очень вкусно!" }


Ответ 201 Created:

{ "id": 99, "user": "me", "text": "Очень вкусно!", "createdAt": "..." }

Поведение на клиенте
При загрузке страницы

Показать спиннер.

Отправить GET на комментарии (offset=0, limit=10).

Отрисовать список.

Если more=false — скрыть кнопку «Ещё».

При нажатии «Ещё»

Увеличить offset на 10.

Подгрузить следующую порцию и добавить в конец списка.

Если more=false — скрыть кнопку.

При отправке комментария

Проверить: текст не пустой, длина например 1..500.

Заблокировать кнопку.

(Опционально) optimistic UI: добавить временный комментарий со статусом “отправляется”.

Отправить POST.

Если успех — заменить временный комментарий на настоящий.

Если ошибка — удалить временный и показать сообщение.

Ошибки

401 Unauthorized → показать “Войдите, чтобы комментировать”.

400 Bad Request → показать текст ошибки валидации.

429 Too Many Requests → “Слишком часто, попробуйте позже”.

500 → “Ошибка сервера”.

Сценарий B — Подписка на автора (AJAX: follow/unfollow)
Контекст

Страница: recipe.html или profile.html.
Есть карточка автора и кнопка подписки.

Цель

Показать текущий статус подписки.

Переключать подписку без перезагрузки.

Обновлять счётчик подписчиков.

UI-элементы

#authorFollowBtn — кнопка

#followersCount — число подписчиков

API
1) Получение статуса подписки

GET /api/users/:authorId/follow

Ответ:

{ "following": false, "followersCount": 100 }

2) Подписаться

POST /api/users/:authorId/follow

Ответ:

{ "following": true, "followersCount": 101 }

3) Отписаться

DELETE /api/users/:authorId/follow

Ответ:

{ "following": false, "followersCount": 100 }

Поведение на клиенте
При загрузке страницы

Запросить GET /follow.

Установить состояние кнопки:

если following=true → “Вы подписаны”

иначе → “Подписаться”

При клике

Если сейчас following=false → отправить POST.

Если following=true → отправить DELETE.

На время запроса — disable кнопку.

После ответа — обновить текст кнопки и счётчик.

Ошибки

401 → предложить авторизацию.

409 → например “нельзя подписаться на самого себя”.

500 → показать toast, вернуть предыдущее состояние.

Сценарий C — Избранное (сердечко: сохранить/удалить рецепт)
Контекст

Страница: recipe.html.
Есть кнопка-иконка “сердечко”.

Цель

При загрузке страницы определить: находится ли рецепт в избранном.

По клику добавлять/удалять рецепт.

Сердечко должно менять цвет:

красное → рецепт в избранном

серое → нет

UI-элементы

#favoriteBtn — кнопка сердечка (или .favorite-btn)

favoriteBtn.classList.contains("active") — признак “красное”

API
1) Проверить статус избранного

GET /api/recipes/:recipeId/favorite

Ответ:

{ "isFavorite": false }

2) Добавить в избранное

POST /api/recipes/:recipeId/favorite

Ответ:

{ "isFavorite": true }

3) Удалить из избранного

DELETE /api/recipes/:recipeId/favorite

Ответ:

{ "isFavorite": false }

Поведение на клиенте
При загрузке

Отправить GET.

Если isFavorite=true → добавить класс active (красное).

Если false → убрать active.

При клике

Если active отсутствует:

optimistic UI: сразу включить active

отправить POST

Если active есть:

optimistic UI: сразу убрать active

отправить DELETE

После ответа

Синхронизировать UI по ответу.

Если ошибка — откатить в предыдущее состояние.

Ошибки

401 → показать “Войдите, чтобы сохранять в избранное”.

500 → откатить сердечко и показать toast.

Сценарий D — Показ состава (ингредиенты) в карточке рецепта через modal
Контекст

Страница: index.html или search.html.
На каждой карточке рецепта есть кнопка «Состав».

Цель

По нажатию кнопки открывать небольшое диалоговое окно (modal) со списком ингредиентов.

Закрывать modal:

при клике на пустую область вне окна

при повторном клике на кнопку «Состав»

(опционально) по Escape

UI-элементы

В каждой карточке:

.recipe-card — карточка

.btn-ingredients — кнопка «Состав»

.ingredients-modal — окно (создаётся динамически или скрытое)

.modal-overlay — затемнение/фон (область клика вне)

API

GET /api/recipes/:recipeId/ingredients

Ответ:

{
  "recipeId": "42",
  "ingredients": [
    { "name": "Яйцо", "amount": 2, "unit": "шт" },
    { "name": "Мука", "amount": 200, "unit": "г" }
  ]
}

Поведение на клиенте
При клике «Состав»

Если modal для этого рецепта закрыт:

открыть modal

показать “Загрузка…”

отправить GET ингредиентов

после ответа — заполнить список

Если modal уже открыт:

закрыть modal (без запросов)

Закрытие по клику вне окна

overlay ловит клик

если клик не внутри окна — закрыть modal

Оптимизация (рекомендуется)

Кэшировать ингредиенты в памяти:

если уже загружали для recipeId — повторно не запрашивать API.

Ошибки

404 → “Рецепт не найден”.

500 → показать “Не удалось загрузить состав”.

Общие требования к реализации AJAX (важно для LMM)

Чтобы LMM писал код правильно, желательно в конце сценариев всегда добавлять:

Использовать fetch() (или axios), JSON.

Все запросы должны быть асинхронными, без перезагрузки страницы.

Для POST/DELETE требуется авторизация:

либо cookie session,

либо Authorization: Bearer <token>.

На время запросов блокировать кнопки, чтобы не было двойных кликов.

Обрабатывать 401/403/500.
