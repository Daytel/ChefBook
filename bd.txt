Каждый сценарий содержит:

Триггер (что делает пользователь / агент).

Последовательность HTTP-вызовов (метод + путь + параметры / тело).

Ожидаемый формат ответа (статус, JSON-схема).

Поведение при ошибках и крайних случаях.

Примеры curl / fetch.

Общая информация по API

Базовый URL (локально): http://localhost:8000

CORS разрешён (*).

Все запросы JSON: Content-Type: application/json и Accept: application/json (для POST).

Основные endpoints:

GET /recipes — список/фильтрация/пагинация.

GET /recipes/{id} — получить рецепт по id.

POST /recipes — создать рецепт.

Поля рецепта (в JSON):

{
  "id": int,                // в ответах сервера (при создании/получении)
  "title": "string",
  "description": "string|null",
  "ingredients": ["string", ...],
  "steps": "string",
  "cuisine": "string|null",
  "difficulty": "easy|medium|hard|null",
  "prep_time": int|null,    // минуты
  "cook_time": int|null,
  "servings": int|null,
  "tags": ["string", ...] | null,
  "created_at": "ISO8601 datetime"
}


Валидные ошибки:

422 Unprocessable Entity — Pydantic валидация поля(ей) (вернёт body с описанием полей и ошибок).

404 Not Found — рецепт не найден (GET /recipes/{id}).

500 — серверная ошибка.

Пагинация/формат ответа GET /recipes:

{
  "total": int,
  "page": int,
  "per_page": int,
  "items": [ {...recipe...}, ... ]
}

Сценарий 1 — Первичный переход: показать все рецепты

Цель: получить первую страницу каталога при загрузке каталога.

Триггер: браузер/агент открывает страницу каталога.

HTTP (single call):

GET /recipes?page=1&per_page=10&sort_by=created_at&sort_dir=desc


Query params (всегда URL-кодировать):

page (int, >=1), per_page (int, 1..100),

sort_by ∈ {created_at,prep_time,title} (по умолчанию created_at),

sort_dir ∈ {asc,desc} (по умолчанию desc).

Ожидаемый ответ:

200 OK + JSON (см. структура пагинации выше).

items — массив рецептов (каждый содержит id, title, ingredients, prep_time, difficulty, cuisine, tags, created_at и пр.).

Крайние случаи / поведение LMM:

Если total == 0 → отобразить «Рецепты не найдены».

Для постраничного отображения: использовать page и per_page чтобы последовательно подгружать страницы (page++ пока есть items.length > 0 или пока page * per_page < total).

Пример curl:

curl -s "http://localhost:8000/recipes?page=1&per_page=10"

Сценарий 2 — Фильтрация по диапазону времени подготовки (min_prep_time / max_prep_time)

Цель: вернуть рецепты с prep_time в заданном диапазоне.

Триггер: пользователь вводит диапазон (от … до …) в UI; фронтенд делает AJAX-запрос.

HTTP:

GET /recipes?min_prep_time=0&max_prep_time=20&page=1&per_page=20


Параметры: min_prep_time (int >=0), max_prep_time (int >=0). Любой из них можно опустить (тогда фильтр по этому краю не применяется).

Ожидаемый ответ:

200 OK + пагинация. items содержит только рецепты с prep_time в указанном диапазоне.

Крайние случаи / поведение LMM:

Если min_prep_time > max_prep_time — не посылать такой запрос; либо поменять местами значения, либо показать ошибку ввода.

Если бэкенд возвращает 200 с пустым items — корректно отобразить "нет результатов".

Для UX: debounce ввода диапазона (LMM/агент не должен швырять запросы при каждом нажатии).

Пример fetch:

fetch('/recipes?min_prep_time=0&max_prep_time=20&page=1')
  .then(r => r.json()).then(json => /* render json.items */)

Сценарий 3 — Фильтрация по кухне (cuisine) или сложности (difficulty)

Цель: отфильтровать рецепты по одному критерию (кухня или сложность).

Триггер: выбор значения в селекте (e.g. cuisine=итальянская или difficulty=easy).

HTTP: пример

GET /recipes?cuisine=итальянская&page=1&per_page=12


или

GET /recipes?difficulty=easy&page=1


difficulty валидация: только easy, medium, hard. Значение вне набора → бэкенд вернёт 422 (проверяется регулярным выражением).

Ожидаемый ответ:

200 OK + пагинация с элементами, удовлетворяющими фильтру.

Крайние случаи / поведение LMM:

Если UI позволяет множественный выбор кухонь/тегов, конкатенируйте запросы по одному фильтру за раз (бэкенд сейчас принимает по одному значению tag/cuisine). Для множественного фильтра можно выполнить несколько запросов и объединить результаты на клиенте, либо расширить API (future).

При получении 422 — извлечь сообщение об ошибке (JSON) и показать юзеру.

Пример curl:

curl -G "http://localhost:8000/recipes" --data-urlencode "cuisine=итальянская" --data "page=1"

Сценарий 4 — Комбинированная фильтрация (несколько критериев одновременно)

Цель: применить >1 фильтр одновременно: (например) cuisine, диапазон prep_time, include_ingredient.

Триггер: пользователь выставляет несколько фильтров; фронтенд отправляет единый GET-запрос с набором параметров.

HTTP (пример):

GET /recipes?cuisine=итальянская&min_prep_time=10&include_ingredient=чеснок&page=1&per_page=20


Доп. параметры:

include_ingredient — строка; сервер делает substring match в JSON-поле ингредиентов. Важно: LMM должен URL-кодировать значение (например, пробелы/кириллица).

exclude_ingredient — аналогично, исключает рецепты, где встречается подстрока.

tag — найти рецепты с данным тегом (поиск по tags_json как по подстроке).

q — полнотекст/substring поиск по title/description/steps.

Ожидаемый ответ:

200 OK + пагинированный набор, уже применённый к нескольким фильтрам.

Крайние случаи / поведение LMM:

Если фильтры конфликтуют → items может быть пустым.

Для больших наборов фильтров и частых запросов — кешировать последний успешный запрос и ответ (ключ кеша = сериализованные параметры запроса).

Если include_ingredient содержит спецсимволы (%, _, & и т. п.) — обязательно URL-кодировать и при необходимости экранировать. (Текущая реализация бэкенда использует LIKE по JSON; это строковый поиск, поэтому частичные совпадения возможны.)

Пример fetch:

const params = new URLSearchParams({
  cuisine: 'итальянская',
  min_prep_time: '10',
  include_ingredient: 'чеснок',
  page: '1'
});
fetch('/recipes?' + params.toString())
  .then(r => r.json())
  .then(json => /* render json.items */);

Сценарий 5 — Добавление нового рецепта (запись)

Цель: создать рецепт в БД; при успешном добавлении фронтенд должен увидеть новый рецепт (либо через повторный GET, либо добавить в текущий список).

Триггер: пользователь заполняет форму add-recipe.html и нажимает «Отправить».

HTTP (single call):

POST /recipes
Content-Type: application/json

{ 
  "title": "Название рецепта",
  "description": "Короткое описание",
  "ingredients": ["яйца","молоко","соль"],
  "steps": "Пошаговая инструкция...",
  "cuisine": "русская",
  "difficulty": "easy",
  "prep_time": 10,
  "cook_time": 5,
  "servings": 2,
  "tags": ["завтрак","быстро"]
}


Валидация (перед отправкой / что ожидает сервер):

title: строка, min_length=3, max_length=200 (обязательное).

ingredients: массив строк, min_items=1, элементы не пустые (обязательное).

steps: строка, min_length=3 (обязательное).

difficulty: если задан — easy|medium|hard (строгая проверка).

prep_time, cook_time — если заданы, должны быть int >= 0.

servings — если задан, int >= 1.

Ожидаемые ответы:

201 Created + JSON нового рецепта (включая id и created_at) — при успехе.

422 Unprocessable Entity + JSON с ошибками — при нарушении валидации (привести поля и причины).

500 — внутренние ошибки.

Дальнейшие шаги (после 201):

Либо агент делает GET /recipes (текущая страница) чтобы подгрузить обновлённый список, либо локально добавляет полученный recipe в UI (лучше — оптимистично добавить, затем подтвердить через GET).

Поведение при ошибках / рекомендации LMM:

Если сервер вернул 422, извлечь структуру ошибок (обычно в теле JSON) и отобразить пользователю (какое поле и почему).

Для повторных попыток: не ретраять POST автоматически без вмешательства пользователя (риск дублей). Вместо этого показать пользователю ошибку и предложить исправить.

Для предотвращения дублирования: сервер возвращает id — при повторной попытке проверять наличие этого id (если клиент переотправляет одно и то же тело, важно иметь idempotency ключ — в текущей API нет; для future версий можно добавить заголовок Idempotency-Key).

Пример curl:

curl -X POST http://localhost:8000/recipes \
  -H "Content-Type: application/json" \
  -d '{"title":"Быстрый омлет","ingredients":["яйца","молоко","соль"],"steps":"Взбить яйца..." }'
